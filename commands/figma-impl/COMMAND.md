# /figma-impl

Figma URL을 받아 해당 디자인을 현재 프로젝트에 구현합니다.

## 옵션

URL 없이 호출하면 작업을 시작하지 않고 URL을 요청합니다.

```
/figma-impl <figma-url>
/figma-impl https://www.figma.com/design/FILE_KEY/...?node-id=NODE_ID
```

---

## 실행 순서

### 1단계: URL 파싱

Figma URL에서 `fileKey`와 `nodeId`를 추출합니다.

- `figma.com/design/:fileKey/...?node-id=:nodeId` 형식에서 추출
- `nodeId`의 `-`를 `:`로 변환 (예: `6216-7502` → `6216:7502`)
- FigJam URL(`figma.com/board/...`)은 `get_figjam` 도구를 사용

### 2단계: 디자인 컨텍스트 수집 (병렬)

```
get_design_context(fileKey, nodeId)   ← 컴포넌트 구조, 스타일, Code Connect 힌트
get_variable_defs(fileKey)            ← Figma 디자인 토큰 (색상, 간격, 타이포그래피 변수)
```

반환된 코드는 React + Tailwind 기반 레퍼런스입니다. 시각적 추측에 의존하지 않고, 반환된 구조와 토큰 값을 기반으로 구현합니다.

### 3단계: 코드베이스 파악 (병렬)

디자인에 등장하는 컴포넌트·스타일과 관련된 기존 코드를 탐색합니다.

- 동일하거나 유사한 컴포넌트가 이미 존재하는지 확인
- 프로젝트의 색상·간격·타이포그래피 토큰 체계 파악
- 공통 UI 컴포넌트 라이브러리 위치 및 사용 패턴 확인
- 레이아웃 컨벤션 (className 구조, 반응형 처리 방식 등)

### 4단계: 구현 계획 수립

구현 전에 아래 항목을 정리합니다.

| 항목 | 내용 |
|------|------|
| 생성할 파일 | 경로 및 컴포넌트명 |
| 재사용할 기존 코드 | 프로젝트 내 기존 컴포넌트·토큰 |
| 주의사항 | 디자인 어노테이션, 제약사항 |

계획이 불명확하거나 기존 컴포넌트와 충돌 가능성이 있으면 구현 전에 확인합니다.

### 5단계: 구현

다음 우선순위로 구현합니다.

1. **기존 코드 재사용** — 새로 만들기 전에 프로젝트에 이미 있는 컴포넌트·토큰을 먼저 활용
2. **프로젝트 컨벤션 준수** — Figma 레퍼런스 코드를 그대로 붙여 넣지 않고 프로젝트 스택에 맞게 변환; 디자인에 아이콘이 필요한 경우 프로젝트에서 사용 중인 아이콘 라이브러리를 확인하여 추가()
3. **타입 안전성** — TypeScript props 타입 명시, nullable 여부 반영

#### 변환 규칙

- `get_variable_defs`로 수집한 Figma 토큰 → 프로젝트 CSS 변수 또는 토큰과 대조하여 매핑
- Figma 토큰에 대응하는 프로젝트 토큰이 없으면, 하드코딩 대신 누락 토큰을 명시하고 확인을 요청
- Figma의 절대 수치(px)는 토큰으로 대체 가능한지 먼저 확인하고, 불가능한 경우에만 수치 사용
- Code Connect 스니펫이 반환된 경우 → 해당 컴포넌트를 직접 사용
- 시각적 추측(스크린샷 참조, 색상 눈대중)으로 값을 결정하지 않음

### 6단계: 구현 결과 보고

```markdown
## Figma 구현 결과

### 생성·수정된 파일

| 파일 경로 | 작업 |
|-----------|------|
| src/components/... | 신규 |
| src/... | 수정 |

### 재사용한 기존 코드

- 컴포넌트: ...
- 토큰/변수: ...

### 확인 필요

- [WARN] ...
```

---

## 유의사항

- `get_design_context` 반환 코드는 레퍼런스일 뿐이며 최종 코드가 아닙니다.
- 디자인 어노테이션이 있으면 반드시 반영합니다.
- 구현 범위가 단일 컴포넌트를 넘어 페이지·레이아웃 전체인 경우, 범위를 먼저 확인합니다.
- 디자인과 기존 코드 사이에 충돌이 있으면 구현 전에 어떻게 처리할지 확인합니다.
