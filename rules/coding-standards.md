# Coding Standards (코딩 기준)

모든 코드 작업에 적용되는 객체지향 원칙과 클린코드 기준입니다.
Claude는 SOLID, Clean Code, DRY, KISS, YAGNI를 이미 알고 있으므로, 프로젝트 체크리스트만 명시합니다.

**참조 시점**:
- Worker: 코드 작성 시작 전 필수 참조
- Reviewer: 코드 품질 검토 시 기준으로 사용

---

## SOLID 원칙 체크리스트

### Single Responsibility (단일 책임)
- [ ] 클래스/함수가 **하나의 변경 이유**만 가지는가?
- [ ] "그리고(and)"로 설명되는 책임이 있는가? → 분리 필요
- [ ] 클래스명이 역할을 명확히 표현하는가?

### Open/Closed (개방-폐쇄)
- [ ] 새 기능 추가 시 **기존 코드 수정 없이** 확장 가능한가?
- [ ] 인터페이스/추상화로 변경에 대비했는가?
- [ ] 조건문(if/switch) 대신 다형성을 고려했는가?

### Liskov Substitution (리스코프 치환)
- [ ] 자식 클래스가 부모 클래스의 **계약(contract)을 위반**하지 않는가?
- [ ] 부모 타입으로 선언된 변수에 자식 타입을 넣어도 정상 작동하는가?
- [ ] 자식 클래스가 부모보다 **더 엄격한 전제조건**을 요구하지 않는가?

### Interface Segregation (인터페이스 분리)
- [ ] 인터페이스가 **너무 많은 메서드**를 강제하지 않는가?
- [ ] 구현체가 **빈 메서드**를 가지지 않는가?
- [ ] 역할별로 인터페이스가 분리되어 있는가?

### Dependency Inversion (의존성 역전)
- [ ] 상위 모듈이 **구체 클래스 대신 인터페이스**에 의존하는가?
- [ ] 의존성이 외부에서 주입(DI)되는가?
- [ ] 구체 클래스를 직접 생성(`new`)하지 않는가?

## 기본 원칙 체크리스트

### DRY (Don't Repeat Yourself)
- [ ] 동일한 로직이 **여러 곳에 복사**되어 있지 않은가?
- [ ] 변경 시 **여러 곳을 수정**해야 하지 않는가?
- [ ] 공통 로직을 함수/클래스로 추출했는가?

### KISS (Keep It Simple)
- [ ] 불필요하게 복잡한 추상화를 사용하지 않았는가?
- [ ] 코드가 **직관적으로 이해**되는가?
- [ ] 과도한 디자인 패턴을 적용하지 않았는가?
- **가이드**: 추상화는 중복이 3번 이상 발생할 때 고려

### YAGNI (You Aren't Gonna Need It)
- [ ] 현재 요구사항에 **실제로 필요한** 기능인가?
- [ ] "나중에 필요할 것 같아서" 추가한 코드가 있는가?
- [ ] 사용되지 않는 매개변수/옵션이 있는가?

## 가독성 체크리스트

### 코드 구조
- [ ] 함수/클래스 길이가 적절한가? (함수 20줄, 클래스 200줄 이내 권장)
- [ ] 중첩 깊이가 **2단계 이내**인가?
- [ ] Early return을 활용해 중첩을 줄였는가?
- [ ] 관련 있는 코드끼리 **그룹핑**되어 있는가?
- [ ] 한 줄에 **하나의 작업**만 수행하는가?

### 명확성
- [ ] 매직 넘버/문자열을 **상수로 추출**했는가?
- [ ] 복잡한 조건문을 **의미 있는 변수/함수로 추출**했는가?
- [ ] 불필요한 else 블록을 제거했는가?
- [ ] 부정문(`!isNotValid`)보다 긍정문(`isValid`)을 사용했는가?
- [ ] 삼항 연산자가 과도하게 중첩되지 않았는가? (1단계까지만)

**예시**:
```
❌ if (user.age >= 18 && user.country === "KR" && user.verified)
✅ const isEligible = user.age >= 18 && user.country === "KR" && user.verified
   if (isEligible)

❌ if (!isNotValid) { ... }
✅ if (isValid) { ... }

❌ return items.length > 0 ? items.filter(x => x.active).length > 5 ? "many" : "few" : "none"
✅ if (items.length === 0) return "none"
   const activeCount = items.filter(x => x.active).length
   return activeCount > 5 ? "many" : "few"
```

### 일관성
- [ ] 코드베이스 전체에서 **동일한 패턴**을 사용하는가?
- [ ] 동일한 개념을 **동일한 용어**로 표현하는가?
  - 예: `get/fetch/retrieve` 중 하나로 통일
- [ ] 파일/폴더 구조가 **일관적**인가?

## Clean Code 체크리스트

### 네이밍
- [ ] 이름만 보고 **역할/목적을 파악**할 수 있는가?
- [ ] `temp`, `data`, `info` 같은 **모호한 이름**을 피했는가?
- [ ] 약어 대신 **전체 단어** 사용 (도메인 표준 약어는 예외)
- [ ] 불필요한 접두사(`a`, `the`) 제거
- [ ] boolean은 `is`, `has`, `can`으로 시작하는가?
- [ ] 컬렉션은 복수형(`users`, `items`)인가?

### 함수
- [ ] 함수가 **한 가지 작업**만 수행하는가?
- [ ] 함수 길이가 **20줄 이내**인가? (권장)
- [ ] 들여쓰기 깊이가 **2단계 이내**인가?
- [ ] 매개변수 개수가 **3개 이하**인가?
- [ ] 부수 효과(side effect)가 명확히 드러나는가?
- [ ] 함수명이 **동사**로 시작하는가? (`calculate`, `validate`, `send`)

### 주석
- [ ] **코드 자체가 설명적**인가? (주석 없이 이해 가능)
- [ ] 주석이 "무엇"이 아닌 **"왜"**를 설명하는가?
- [ ] 주석이 코드와 **동기화**되어 있는가?
- [ ] 주석 처리된 코드(commented-out code)가 없는가?

**필요한 주석만**:
- 복잡한 알고리즘의 의도
- 외부 제약사항 (API 제한, 표준 스펙)
- 경고 (성능, 보안, 부작용)
- TODO (임시 해결책, 향후 개선 사항)

### 에러 처리
- [ ] 예외 상황이 **명시적으로 처리**되는가?
- [ ] 에러 메시지가 **구체적**인가?
- [ ] 빈 catch 블록이 없는가?
- [ ] null 체크가 적절히 수행되는가?

### 포맷팅
- [ ] 프로젝트의 **코딩 컨벤션**을 따르는가?
- [ ] 린터(ESLint, Prettier 등) 설정을 통과하는가?
- [ ] 들여쓰기가 일관되는가?
- [ ] 파일 길이가 적절한가? (500줄 이내 권장)

## 객체지향 설계 체크리스트

### Encapsulation (캡슐화)
- [ ] 데이터(필드)가 외부에 **직접 노출되지 않는가**?
- [ ] getter/setter가 **단순 접근 이상의 로직**을 포함하는가?
- [ ] 객체가 스스로 **행동을 결정**하는가?

### Composition over Inheritance (조합 우선)
- [ ] "is-a" 관계가 **명확한가**? (아니면 조합 사용)
- [ ] 상속 깊이가 **2단계 이내**인가?
- [ ] 부모 클래스 변경이 자식에게 **부작용**을 주지 않는가?

### Tell, Don't Ask (묻지 말고 시켜라)
- [ ] 객체의 상태를 묻지 말고, 행동을 시키는가?
- 예: `if (user.getBalance() > 100) { user.setBalance(...) }` ❌
- 예: `user.deduct(100)` ✅

### Boundaries (경계)
- [ ] 외부 라이브러리를 **래퍼 클래스**로 감쌌는가?
- [ ] 외부 API 응답을 **내부 모델로 변환**하는가?
- [ ] 타입 변환이 경계에서 수행되는가?

## 보안 체크리스트

- [ ] 비밀번호, API 키 등을 **하드코딩**하지 않았는가?
- [ ] 환경 변수 또는 비밀 관리 시스템 사용
- [ ] 사용자 입력을 **검증(validation)**하는가?
- [ ] SQL 인젝션, XSS 등 OWASP Top 10 취약점 확인

---

## 성능 체크리스트

- [ ] 불필요한 루프/연산을 제거했는가?
- [ ] 데이터베이스 N+1 쿼리 문제가 없는가?
- [ ] 적절한 자료구조를 선택했는가? (배열 vs 해시맵)
- [ ] 조기 최적화를 피했는가? (측정 후 최적화)

---

## 테스트 가능성 체크리스트

- [ ] 함수가 **순수 함수** 형태인가? (같은 입력 → 같은 출력)
- [ ] 의존성이 **주입 가능**한가?
- [ ] 부수 효과가 **최소화**되었는가?
- [ ] 테스트하기 어려운 로직이 분리되었는가? (시간, 랜덤, 외부 API)

---

## Reviewer 평가 기준

### 필수 준수 항목 (하나라도 미달 시 재작업)

- [ ] SOLID 원칙 위반 **없음**
- [ ] DRY 원칙: 중복 로직 **없음**
- [ ] 의미 있는 이름 사용
- [ ] 함수 단일 책임 준수
- [ ] 에러 처리 적절
- [ ] 보안 이슈 **없음** (하드코딩된 비밀 정보, 입력 검증 누락)
- [ ] 빈 catch 블록 **없음**
- [ ] 주석 처리된 코드 **없음**

### 권장 사항 (개선 제안 가능)

- [ ] KISS 원칙 준수
- [ ] YAGNI 원칙 준수
- [ ] 주석 품질 (왜를 설명하는가)
- [ ] 테스트 가능성
- [ ] 포맷팅 일관성

## 예외 처리

**원칙을 깨도 되는 경우**:
- 성능이 **critical**한 경우 (측정 데이터 기반)
- 레거시 코드와의 **호환성** 필요
- 외부 라이브러리 API **제약**
- 프로토타입/POC

**단, 반드시 주석으로 이유 명시**:
```
// PERFORMANCE: 루프 언롤링으로 10배 속도 향상 (벤치마크: ...)
// TODO: 추후 가독성 개선 고려
for (let i = 0; i < n; i += 4) { ... }
```
