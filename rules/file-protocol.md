# 파일 기반 통신 프로토콜

서브에이전트 간 통신에 사용되는 파일 형식 정의입니다.
모든 에이전트가 이 형식을 따라 파일을 읽고 씁니다.

---

## 파일 통신 흐름

```
Step 1: Strategist → SPEC.md
            ↓
Step 2: Plan-Reviewer → PLAN.md (reads SPEC.md)
            ↓
Step 4: Secretary → User approval (reads SPEC.md)
            ↓
Step 5: Worker → WORK_LOG.md (reads SPEC.md)
            ↓
Step 6: Reviewer → QA_REPORT.md (reads SPEC.md, WORK_LOG.md)
            ↓
Step 7: Secretary → Decision (reads QA_REPORT.md)
```

---

## SPEC.md (계획 문서)

**생성자**: Strategist
**독자**: Plan-Reviewer, Worker, Reviewer, Secretary

### 형식

```markdown
# 프로젝트 계획서

## 1. 문제 분석

### 표면적 요청
[사용자가 요청한 내용 그대로]

### 근본 문제
[왜 이 요청을 하는가? 해결하려는 본질적 문제]

### 제약 조건
- 기술 스택: [사용 가능한/불가능한 기술]
- 시간: [마감 또는 우선순위]
- 범위: [포함/제외 사항]

## 2. 맥락 파악

### 기존 시스템
[현재 코드베이스 상태, 관련 파일]

### 영향 범위
[이 작업이 영향을 주는 부분]

### 의존성
[필요한 라이브러리, 서비스, 설정]

## 3. 접근 방법

### 옵션 1: [방법명]
- 장점: [...]
- 단점: [...]
- 트레이드오프: [...]

### 옵션 2: [방법명]
- 장점: [...]
- 단점: [...]
- 트레이드오프: [...]

### 선택한 방법: [옵션 N]
**이유**: [왜 이 방법을 선택했는가]

## 4. 실행 계획

### 목표
[한 줄 요약: 무엇을 달성할 것인가]

### 단계
1. [Worker] — [작업 내용]
2. [Worker] — [작업 내용]
3. [Task(Infra-Engineer)] — [작업 내용]

### 예상 결과물

- 설정: [추가될 설정]

### 검증 기준
- [ ] [검증 항목 1]
- [ ] [검증 항목 2]
- [ ] [검증 항목 3]

## 5. 리스크 및 대응

### 예상 리스크
1. [리스크 1]: [발생 가능성, 영향도]
   - 대응: [어떻게 완화/회피할 것인가]
2. [리스크 2]: ...

### 롤백 방법
[문제 발생 시 원래 상태로 되돌리는 방법]
```

### 필수 섹션
- ✅ 문제 분석 (표면적 요청, 근본 문제)
- ✅ 접근 방법 (여러 옵션 비교, 선택 이유)
- ✅ 실행 계획 (단계, 예상 결과물, 검증 기준)
- ✅ 리스크 및 대응

## PLAN.md (계획 검토 결과)

**생성자**: Plan-Reviewer
**독자**: Secretary

### 형식

```markdown
# 계획 검토 결과

## 판정
[승인 / 재수립 필요]

## 검토 요약
[한 줄 요약: 계획의 전반적 평가]

## 상세 검토

### 1. Secretary 역할 준수 여부 ⭐ CRITICAL
- [✅ / ❌] Secretary가 코드/인프라/문서를 직접 작성하지 않음
- [✅ / ❌] 모든 실행 작업이 Worker에게 위임됨
- [✅ / ❌] 파일 쓰기가 Worker/Reviewer로 배정됨

**문제점**: [있으면 작성]

### 2. 계획 완정성
- [✅ / ❌] 문제 분석이 충분함
- [✅ / ❌] 접근 방법이 여러 옵션 비교됨
- [✅ / ❌] 실행 계획이 구체적임
- [✅ / ❌] 검증 기준이 명확함

**문제점**: [있으면 작성]

### 3. 에이전트 배정 적절성
- [✅ / ❌] Worker 작업이 명확히 정의됨
- [✅ / ❌] 전문 Agent 호출이 적절함 (복잡한 작업 → TypeScript-Engineer, Infra-Engineer 등)
- [✅ / ❌] 에이전트 역할 분리가 올바름

**문제점**: [있으면 작성]

### 4. 실행 가능성
- [✅ / ❌] 단계가 순서대로 실행 가능
- [✅ / ❌] 의존성이 올바르게 처리됨
- [✅ / ❌] 리스크 대응이 현실적임

**문제점**: [있으면 작성]

## 재수립 지시 (재수립 필요 시)
1. [구체적 수정 사항 1]
2. [구체적 수정 사항 2]
...

## 승인 사유 (승인 시)
[왜 이 계획이 실행 가능하다고 판단했는가]
```

### 필수 항목
- ✅ 판정 (승인/재수립 필요)
- ✅ Secretary 역할 준수 검토 (CRITICAL)
- ✅ 4가지 검토 항목
- ✅ 재수립 지시 (재수립 시) 또는 승인 사유 (승인 시)

## 파일 작성 규칙

### 모든 파일 공통

1. **Markdown 형식**
   - 명확한 계층 구조 (H1, H2, H3)
   - 코드 블록 사용 (```bash, ```markdown 등)

2. **구체성**
   - 추상적 표현 금지
   - 파일명, 라인 번호, 로그 포함

3. **완결성**
   - 필수 섹션 모두 포함
   - 빈 섹션 없음

### Secretary 읽기 규칙

**판정 확인**:
```python
# PLAN.md 읽기
if "승인" in PLAN.md:
    proceed_to_step_4()
elif "재수립 필요" in PLAN.md:
    retry_strategist()

# QA_REPORT.md 읽기
if "✅ 승인" in QA_REPORT.md:
    proceed_to_step_8()
elif "❌ 재작업 필요" in QA_REPORT.md:
    retry_worker()
```

### 에이전트 쓰기 규칙

**저장 전 확인**:
- [ ] 필수 섹션 모두 작성
- [ ] 판정/결과 명확히 명시
- [ ] 구체적 내용 (추상적 표현 금지)
- [ ] Markdown 형식 올바름

## 파일 버전 관리

### 재작업 시

**기존 파일 보존**:
```bash
# 1회차
SPEC.md

# 2회차 (재수립)
SPEC_v2.md  # 새 버전
SPEC.md     # 이전 버전 유지 (참고용)

# 최종 확정 후
SPEC.md     # 최종 버전으로 덮어쓰기
```

**또는 누적 방식**:
```bash
# SPEC.md 내부에 변경 이력 포함
## 변경 이력
- v1 (2024-03-31): 초기 계획
- v2 (2024-03-31): Plan-Reviewer 피드백 반영
- v3 (2024-03-31): 접근 방법 변경
```

## Best Practices

1. **읽는 사람 관점**
   - Reviewer가 SPEC.md를 읽을 때 맥락 파악 가능하게
   - Secretary가 PLAN.md를 읽을 때 판정 즉시 확인 가능하게

2. **검증 가능성**
   - 모든 주장에 증거 포함
   - "테스트 통과" → 로그 첨부

3. **일관성**
   - 파일 간 정보 일치
   - SPEC.md의 목표 = WORK_LOG.md의 작업 = QA_REPORT.md의 검토 대상
