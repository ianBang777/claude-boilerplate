# Review Uncommitted Changes

현재 커밋되지 않은 변경사항(unstaged + staged + untracked)을 리뷰하고 문제점과 개선사항을 제시합니다.

## 목적

- 커밋 전에 위험 요소를 사전 차단합니다.
- 리뷰 범위를 "현재 작업 트리"로 한정해 빠르게 피드백합니다.

## 실행 순서

1. 변경 파일 목록을 확인합니다.
   - `git status --short`
2. 실제 코드 변경을 확인합니다.
   - `git diff`
   - `git diff --staged`
3. 신규 파일(untracked)이 있으면 파일 내용을 함께 검토합니다.
4. 아래 체크리스트 기준으로 이슈를 분류합니다.
5. 심각도별 결과와 수정 방향을 제시합니다.

## 심각도 분류

- **[BLOCK]** 반드시 수정해야 커밋 가능 (기능 오동작, 보안/데이터 손실 위험)
- **[WARN]** 수정을 강력 권고 (성능, 유지보수성, 예외 처리 누락)
- **[SUGGEST]** 선택적 개선 (가독성, 일관성, 테스트 보강)

## 체크리스트

### 정확성
- [ ] 요구사항과 다른 동작 가능성은 없는가?
- [ ] null/undefined/빈 값/경계값 처리가 되어 있는가?
- [ ] 비동기 실패/타임아웃/재시도 시나리오가 고려되었는가?

### 타입 및 안정성
- [ ] 타입 단언 남용(`as any`, `@ts-ignore`, non-null assertion)이 없는가?
- [ ] 함수/훅의 반환 타입과 에러 처리가 일관적인가?

### 보안
- [ ] 민감 정보(토큰, 키, 비밀번호)가 코드/로그에 노출되지 않는가?
- [ ] 입력값 검증/권한 검증이 필요한 지점이 누락되지 않았는가?

### 성능
- [ ] 불필요한 반복 연산/리렌더링/쿼리 호출이 없는가?
- [ ] 애니메이션/인터랙션이 UX와 성능 균형을 지키는가?

### 일관성
- [ ] 프로젝트 컨벤션(절대 경로 import, 함수 선언 방식 등)을 준수하는가?
- [ ] 도메인/기능 단위로 응집도 있게 구성되었는가?

### 테스트
- [ ] 변경으로 깨질 수 있는 핵심 플로우 테스트가 있는가?
- [ ] 테스트가 없다면 최소 검증 시나리오를 제시했는가?

## 출력 형식

```
## 리뷰 결과

### [BLOCK] 제목
파일: path/to/file
이유: ...
수정 방향: ...

### [WARN] 제목
파일: path/to/file
이유: ...
수정 방향: ...

### [SUGGEST] 제목
파일: path/to/file
이유: ...
수정 방향: ...

---
총평: 커밋 전 반드시 반영할 항목 / 선택 반영 항목 요약
```

## 원칙

- 사실 기반으로만 지적합니다. 추측은 "가정"으로 명시합니다.
- 이슈가 없으면 "치명적 이슈 없음"을 명확히 말합니다.
- 리뷰 대상 범위는 "현재 커밋되지 않은 변경사항"으로 제한합니다.
