# Code Review

변경된 코드를 리뷰하고 문제점과 개선사항을 제시합니다.

## 실행 순서

1. `git diff master(main)...HEAD` 또는 지정된 파일을 분석합니다.
2. 아래 체크리스트 순서로 검토합니다.
3. 심각도별로 분류해 보고합니다.

## 심각도 분류

- **[BLOCK]** 반드시 수정해야 배포 가능 (버그, 보안, 데이터 손실 위험)
- **[WARN]** 수정을 강력 권고 (성능, 유지보수성, 잠재적 버그)
- **[SUGGEST]** 선택적 개선 (코드 품질, 가독성)

## 체크리스트

### 정확성
- [ ] 엣지 케이스가 처리되어 있는가? (null, undefined, 빈 배열, 0)
- [ ] 비동기 에러 처리가 되어 있는가?
- [ ] 경쟁 조건(race condition) 가능성이 있는가?

### 타입 안전성 (TypeScript)
- [ ] `as any`, `@ts-ignore`, `!` 남용이 없는가?
- [ ] API 응답 타입이 명시되어 있는가?
- [ ] 옵셔널 체이닝이 적절히 사용되었는가?
- [ ] 타입 단언 사용 금지
- [ ] 절대 경로로만 사용(상대 경로 금지)

### 보안
- [ ] 사용자 입력이 검증·이스케이프 되었는가?
- [ ] 민감한 정보(토큰, 키)가 노출되지 않는가?
- [ ] 권한 체크가 적절히 이루어지는가?

### 성능
- [ ] 불필요한 리렌더링이 발생하지 않는가? (FE)
- [ ] N+1 쿼리 문제가 없는가? (BE)
- [ ] 대용량 데이터에서 메모리 문제가 없는가?

### 유지보수성
- [ ] 함수·컴포넌트가 단일 책임을 지키는가?
- [ ] 중복 코드가 없는가?
- [ ] 로직이 명확하지 않은 부분에 주석이 있는가?

## 출력 형식

```
## 리뷰 결과

### [BLOCK] 제목
파일: src/...
이유: ...
수정 방향: ...

### [WARN] 제목
...

### [SUGGEST] 제목
...

---
총평: ...
```
